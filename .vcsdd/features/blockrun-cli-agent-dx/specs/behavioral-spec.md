## 0. Scope

### Non-goals (REQ-DX-NG-*)

- REQ-DX-NG-001: This feature SHALL NOT change any money-path behavior — payment signing (x402),
  budget arithmetic (`checkBudget`/`reserveBudget`/`recordActualSpend`), or which upstream endpoint a
  command calls. It only ADDS classification/labeling around outcomes that already occur today.
- REQ-DX-NG-002: This feature SHALL NOT change any EXISTING successful (`ok()`-path) JSON output shape
  for the 18 pre-existing commands — new fields may be ADDED (e.g. `balanceUnavailableReason`), but no
  existing field may be renamed, removed, or change type/meaning.
- REQ-DX-NG-003: This feature SHALL NOT remove or change the meaning of the existing error JSON's
  `message` field — the new `code` field (REQ-DX-011) is PURELY additive, preserving backward
  compatibility for any existing caller that only reads `message`. The ONE narrow exception is
  `network_error` (REQ-DX-015): `message` gains an appended network-failure marker for that ONE
  previously-uninformative failure class (today's raw `"fetch failed"` carries zero specific detail) —
  this is enrichment of an empty signal, not a change to existing meaningful content, so it does not
  violate this REQ's "preserve backward compatibility" intent for any caller parsing `message` for
  content it could have relied on before this feature existed.
- REQ-DX-NG-004: This feature SHALL NOT hand-maintain a duplicate flag/parameter catalog anywhere —
  `blockrun commands`'s flag metadata (REQ-DX-003/004) SHALL be generated by introspecting each
  command's REAL zod `schema` export (`src/args/<command>.ts`) at runtime, never a separately
  -maintained list that could silently drift from the real schema.
- REQ-DX-NG-005: This feature SHALL NOT add a 19th PAID command — `blockrun commands` is a free,
  local-only introspection command (no wallet, no network call), exactly like `models`/`dex` today.

## 1. `blockrun commands` — machine-readable command catalog (P0-2)

- REQ-DX-001: THE SYSTEM SHALL add a new top-level subcommand `blockrun commands`, registered
  alongside (not replacing) the 18 existing subcommands. It SHALL make NO wallet call and NO network
  call under any circumstances (REQ-DX-NG-005) — verified by the same absence-of-`gatePaidCall`
  pattern that already marks `wallet`/`models`/`dex` as free (confirmed via `grep -c gatePaidCall
  src/commands/{wallet,models,dex}.ts` → `0` for all three, vs `2` for the other 15 — this is the
  REAL, mechanically-verified ground truth this REQ's implementation must match).
- REQ-DX-002: `blockrun commands --json` SHALL emit a SINGLE JSON document on stdout of shape
  `{ "commands": [ {...}, ... ] }`, containing EXACTLY 18 entries — one per EXISTING subcommand
  (`wallet` through `surf`, the same 18 names `node dist/index.js --help`'s `Commands:` block lists) —
  and SHALL NOT include an entry for `commands` itself (an MCP `tools/list` equivalent lists the real
  callable tools, not a meta-tool about itself; `blockrun-mcp` has no analogous concept since MCP's own
  protocol handles `tools/list`, so there is nothing for `commands` to "mirror" here — this is a
  CLI-only, deliberately self-excluding introspection command).
- REQ-DX-003: Each catalog entry SHALL have the shape `{ name, description, costModel, flags }`:
  - `name`: the real subcommand name (string), e.g. `"video"`.
  - `description`: VERBATIM the SAME string already passed to that command's `.description(...)` call
    in `src/index.ts` (the exact text shown in `blockrun <name> --help`'s output) — a single source of
    truth, zero risk of the catalog's description drifting from the real `--help` text, since it IS
    that text, read at runtime, not copied.
  - `costModel`: EITHER `"free"` or `"paid"`, derived from whether that command's `src/commands/
    <name>.ts` calls `gatePaidCall` at all (per REQ-DX-001's grep-verified ground truth: `wallet`,
    `models`, `dex` → `"free"`; the other 15 → `"paid"`). THE SYSTEM SHALL NOT hand-maintain this
    free/paid list separately — Phase 2 SHALL derive it from the SAME real fact this REQ cites (e.g. a
    small generated/derived registry built from that fact, not an independently-typed-out list that
    could drift).
  - `flags`: an array, one entry per TOP-LEVEL field in that command's zod `schema` export from
    `src/args/<name>.ts`, per REQ-DX-004.
- REQ-DX-004: Each `flags[]` entry SHALL be `{ flag, type, required, enum?, default? }`, derived via
  RUNTIME introspection of the zod schema (never hardcoded), per this EXACT, directly-verified zod
  `4.4.3` API surface (confirmed live in this repo's Node REPL before writing this REQ):
  - `flag`: the kebab-case CLI flag form of the zod field's snake_case name (e.g. zod field
    `duration_seconds` → `"--duration-seconds"`), per the ALREADY-ESTABLISHED convention from the base
    CLI's own behavioral-spec.md REQ-004a ("every CLI flag is the kebab-case form of the source tool's
    exact zod field name") — this feature does NOT invent a new naming rule, it reuses the existing one
    verbatim for the introspector's output.
  - `type`: one of `"string"|"number"|"boolean"|"enum"|"array"|"object"|"any"`, read from the zod
    field's `_def.type` AFTER unwrapping any `ZodOptional`/`ZodDefault` wrapper (confirmed live: a
    `.optional()` field has `constructor.name === "ZodOptional"`, `_def.type === "optional"`, and
    `_def.innerType` holding the real underlying schema node; a `.optional().default(x)` field is
    `ZodDefault` with `_def.type === "default"`, `_def.innerType` holding the (already-optional) inner
    schema, and `_def.defaultValue` holding the literal default VALUE directly, not a function —
    confirmed live: `z.number().optional().default(1024)._def.defaultValue === 1024`).
  - `required`: `!field.isOptional()` (zod's own `.isOptional()` method, confirmed live to correctly
    report `false` for a bare required field and `true` for any `.optional()`/`.optional().default()`
    field).
  - `enum`: present ONLY when `type === "enum"` (i.e. the unwrapped field is `ZodEnum`), the EXACT
    array from that field's `.options` property (confirmed live: `z.enum(["price","history","list"])
    .options` → `["price","history","list"]`, a simpler and equally-real API than parsing the lower
    -level `_def.entries` object).
  - `default`: present ONLY when the field is wrapped in `ZodDefault` (i.e. `.default(...)` was
    called), the exact `_def.defaultValue` (confirmed a direct value, not a function, for every
    default value actually used across the 18 schemas — all of which are plain numbers/strings/
    booleans/enum-strings, never a computed default).
- REQ-DX-005: `blockrun commands` (WITHOUT `--json`) SHALL render a human-readable TABLE — one row per
  command, columns `command | description | cost` (the `costModel` value, or a short "Free"/"Paid"
  label) — mirroring the SAME data as `--json`'s `commands[]` array MINUS the per-flag detail (a
  human wanting flag detail already has `blockrun <cmd> --help` for that; keeping `commands`'s own
  default output concise is the point of the table view).
- REQ-DX-006: `blockrun commands --help` SHALL behave like every other command's `--help` (Commander's
  built-in behavior) — no special-casing required beyond normal subcommand registration.
- REQ-DX-007: THE catalog (both `--json` and human-table forms) SHALL be computed FRESH on every
  invocation, by introspecting the ALREADY-LOADED `schema` exports and reading the ALREADY-DEFINED
  `.description(...)` strings at runtime — NEVER from a build-time-generated static file or a cached
  snapshot — so a future change to any of the 18 commands' zod schema or Commander description is
  reflected in `blockrun commands`'s output automatically, with zero risk of the catalog silently
  going stale (this is the concrete mechanism that satisfies REQ-DX-NG-004's "no hand-maintained
  duplicate" requirement).
- REQ-DX-008: `blockrun commands`'s OWN registration in `src/index.ts` SHALL follow the SAME
  `withCommon`/pattern as the other 18 commands where applicable (`--json` flag support per REQ-DX-002)
  — but SHALL NOT gain `--budget-limit` or `--agent-id` (both are meaningless for a free, no-network
  command; omitting them is itself the correct, minimal design, not an oversight).

## 2. Machine-readable error codes + exit code taxonomy (P0-3)

- REQ-DX-010: `--json` error output SHALL gain a NEW field `code`: the shape becomes
  `{ "error": true, "code": "<machine_code>", "message": "<existing human text>" }` — WHEN a `code`
  applies (REQ-DX-011); the EXISTING `message` field's content and meaning SHALL NOT change (REQ-DX
  -NG-003) — today's `formatError()`-derived human text remains exactly what `message` carries.
- REQ-DX-011: THE `code` taxonomy SHALL be EXACTLY these 6 values, each grounded in an ALREADY
  -REAL, currently-distinguishable failure class in this codebase (verified by reading the actual
  source, not invented). **Classification order (added per spec-review it-1 SPEC-DX-1/SPEC-DX-3 —
  mechanism specified in REQ-DX-015/REQ-DX-016 below)**: items 1-3 below (`usage_error`/
  `budget_exceeded`/`quote_exceeded`) are detected at STRUCTURALLY distinct points in the code
  (pre-network validation, pre-network budget gate, post-quote pre-signature abort) and can never
  collide with each other or with items 4-6. For an error that reaches the generic post-network
  catch-and-classify point (i.e. NOT already one of items 1-3), THE SYSTEM SHALL check, IN THIS
  ORDER, stopping at the first match: (a) `network_error` (item 6 — checked FIRST, because a
  transport-level failure that never received an HTTP response cannot also contain a `"balance"`/
  `"500"`/etc. token, so there is no real ambiguity in checking it first); (b) REQ-DX-016's
  order-preserving extraction of `formatError()`'s EXISTING branch order —
  `isModelUnavailable` → `isServerError` → `isPaymentError` (mapping to `upstream_error`/
  `upstream_error`/`insufficient_funds` respectively); (c) no match → `code` omitted (the fallback).
  1. `"usage_error"` — a `buildRequest()` `{ ok: false, error }` validation failure (malformed,
     missing, or conflicting CLI input — e.g. `src/args/rpc.ts`'s `"--network is required"` /
     `"Invalid network '...'. Use a chain slug like 'ethereum', 'base', or 'solana'."`, or
     `src/args/shared.ts`'s `resolvePositionalAlias`'s conflict error) — detected in EVERY
     `commands/<name>.ts` BEFORE any network call and BEFORE any budget check (the
     `if (!built.ok) return fail(built.error, opts.json);` line present at the top of every
     command handler, verified present in `defi.ts`/`video.ts`/etc.).
  2. `"budget_exceeded"` — a `gatePaidCall()` pre-call rejection: EITHER the ephemeral per-invocation
     `--budget-limit` cap OR the persisted `~/.blockrun/cli-budget.json` global/agent cap — detected in
     `src/commands/shared.ts`'s `gatePaidCall`, BEFORE any network call, AFTER validation passes.
     Ground-truth message patterns (verbatim from `src/core/budget.ts`'s `checkBudget`, reused by the
     persisted-ledger check via `toBudgetState`): `` `Global budget limit $X would be exceeded (...)` ``
     and `` `Agent "<id>" budget $X would be exceeded (...)` ``.
  3. `"quote_exceeded"` — a REAL 402-quoted price rejected AFTER a network round-trip (the quote
     itself required one call) but BEFORE any payment signature is produced. Two ground-truth message
     patterns, BOTH classified to this ONE code: `src/commands/video.ts`'s explicit
     `` `Quote $X exceeds --max-quote-usd $Y — aborting before signing.` `` (the CLI-only
     `--max-quote-usd` gate), and `src/commands/shared.ts`'s `reverify`'s
     `"Budget cap would be exceeded by the real quoted price."` (the general REQ-220 re-check gate
     used by video/music/speech/realface/Solana-image).
  4. `"insufficient_funds"` — `src/core/errors.ts`'s `formatError()`'s EXISTING `isPaymentError`
     classification (message contains an HTTP `402` token, or `"balance"`/`"insufficient"`, or
     `"payment"` without an accompanying `500`) — a REAL wallet-balance rejection from an ACTUAL x402
     settlement attempt (as opposed to `quote_exceeded`'s pre-signature abort, or `budget_exceeded`'s
     pre-network local cap).
  5. `"upstream_error"` — `formatError()`'s EXISTING `isModelUnavailable` (message contains
     `"not active for requested provider"` or `"not found or not active"`) OR `isServerError`
     (HTTP `500`, or `"api error after payment"`) classification.
  6. `"network_error"` — a raw connection/transport failure that occurs BEFORE any HTTP response is
     received at all. Ground truth, directly verified live in this repo (`node -e` against a real
     unreachable host/port) before writing this REQ, per spec-review it-1 SPEC-DX-1: Node's `fetch()`
     throws a `TypeError` with `.name === "TypeError"` and `.message === "fetch failed"`, whose
     `.cause` is a real `Error` carrying a `.code` matching `ECONNREFUSED`/`ENOTFOUND`/`ETIMEDOUT`/
     `EAI_AGAIN` (confirmed for both DNS-not-found and connection-refused cases). For a client-side
     timeout, the error's `.name` is `"TimeoutError"` when triggered via `AbortSignal.timeout()`
     (confirmed live — NOT `"AbortError"`, correcting this spec's own earlier draft, which assumed
     `"AbortError"` without having verified it) — but `src/shell/http.ts`'s `fetchWithTimeout()` (used
     by `manual-x402.ts`/`solana-x402.ts`, i.e. the `video`/`music`/`speech`/`realface`/Solana-`image`
     paid flows) uses a manual `AbortController` instead, whose abort can surface as either name
     depending on the code path. Rather than re-deriving this detection, THE SYSTEM SHALL REUSE the
     ALREADY-EXISTING, ALREADY-CORRECT `src/shell/http.ts`'s exported `isTimeoutError(err)` function
     (verified: it already checks BOTH `"AbortError"` and `"TimeoutError"` names, plus message
     -substring fallbacks for `"abort"`/`"timeout"`/`"timed out"`/`"did not complete within"`) — see
     REQ-DX-015 for how this and the `cause.code` check reach the classifier. This is the ONE
     genuinely NEW classification `code` this feature introduces (today's code has NO distinct
     network-failure `code`/label anywhere — a raw fetch failure currently just becomes a generic,
     unclassified error) — grounded in Node's own directly-verified `fetch()` failure shape and this
     repo's own already-existing timeout-detection utility, neither invented.
  - **Fallback**: any error NOT matching one of the 6 patterns above SHALL have `code` OMITTED from the
    JSON entirely (there is NO 7th "unknown_error"/catch-all value) — preserving today's exact
    `{ error: true, message }` shape for any failure this feature does not explicitly classify, which
    also matches the base CLI's own original REQ-010 (`behavioral-spec.md` of the `blockrun-cli`
    feature) which already specified `code` as OPTIONAL (`"code"?: string`) — this feature is the
    first to actually populate it, for exactly these 6 real cases.
- REQ-DX-012: Process exit codes SHALL follow this mapping (gh/curl-inspired convention, per Dais's
  explicit instruction):
  - `0` = success (the EXISTING `ok()` path — UNCHANGED).
  - `2` = usage (`code` is `"usage_error"` OR `"budget_exceeded"` — both are local, pre-network,
    caller-side problems: a malformed invocation, or a caller-configured spending cap being hit;
    grouped together because NEITHER reflects an upstream/payment-settlement/network problem, and
    BOTH are things the caller can fix by changing how they invoke the command).
  - `3` = payment/balance (`code` is `"insufficient_funds"` OR `"quote_exceeded"` — both are
    fundamentally about a payment AMOUNT or a real-money settlement outcome).
  - `4` = upstream/network (`code` is `"upstream_error"` OR `"network_error"`).
  - `1` = anything else — the fallback/no-`code` case, and any currently-uncaught exception path
    (`src/index.ts`'s top-level `.catch()` and `dispatch()`'s `.catch()`) — this is EXACTLY today's
    universal exit code (every `fail()` call currently returns `exitCode: 1` unconditionally), so `1`
    remains the safety-net default whenever a MORE SPECIFIC code (2/3/4) does not apply, not a
    regression from today's behavior for unclassified failures.
- REQ-DX-013: The NON-`--json` (human) error path's stderr text SHALL be UNCHANGED for every failure
  class EXCEPT `network_error` (REQ-DX-NG-003's spirit extended to the human path) — the `code`/
  exit-code system is additive on the machine-readable side and does not alter what a human running
  the CLI directly sees on stderr for `usage_error`/`budget_exceeded`/`quote_exceeded`/
  `insufficient_funds`/`upstream_error`/no-code cases. The ONE exception is `network_error`: because
  `formatError()` renders `` `Error: ${message}` `` verbatim, and REQ-DX-015 appends its
  network-failure marker INTO `message` (via `extractErrorMessage()`) BEFORE `formatError()` ever
  runs, the human stderr text for a genuine network failure gains that marker too. This is NOT a
  behavior change for any PREVIOUSLY-informative case — a raw network failure had NO specific human
  -readable detail before this feature (just the bare, unhelpful `"fetch failed"`), so enriching it is
  additive, not a change to existing meaningful text, consistent with REQ-DX-NG-003.
- REQ-DX-014: EVERY one of the 18 pre-existing commands' error paths SHALL route through ONE SHARED
  classification function (extending `src/core/render.ts`'s `fail()`/`renderError()`, or a new function
  they call) — NOT reimplemented separately inside each `commands/<name>.ts` — so all 18 commands gain
  the SAME `code`/exit-code behavior from ONE change, with no risk of 18 independently-drifting
  copies. **Clarification (added per spec-review it-1 SPEC-DX-1)**: this REQ's "one shared
  classification function" operates on the STRING each of the 18 catch blocks ALREADY produces via
  `extractErrorMessage(err)` before calling `fail(...)` (verified: every `commands/<name>.ts`'s catch
  block is `const msg = extractErrorMessage(err); return fail(msg, opts.json, ...);` or the direct
  `fail(extractErrorMessage(err), opts.json)` equivalent — zero of the 18 catch sites need to change).
  This is possible ONLY because REQ-DX-015 makes `extractErrorMessage()` itself preserve the
  network-failure signal that today it silently discards — the classification function this REQ
  describes still never touches the raw `err` object directly at any of the 18 call sites; it
  classifies the (now-enriched) STRING, exactly like `formatError()` already does today for every
  other case.
- REQ-DX-015 (closes spec-review it-1 SPEC-DX-1 — the network_error classification's ONLY viable
  wiring, per the adversary's suggested option (a)): `src/core/errors.ts`'s `extractErrorMessage(err)`
  SHALL be extended so that, BEFORE returning its string, it checks the RAW `err` object for a
  network-failure signal and — WHEN found — appends a fixed, machine-detectable marker to the
  returned string (exact marker format left to Phase 2, e.g. a suffix like
  `` ` (network:${code})` `` or `` ` (network:timeout)` ``; the REQUIREMENT is that the marker is
  present, greppable, and distinguishable from any existing message content, not the literal string).
  The detection SHALL be, in order: (a) IF `err` is an `object` with a `cause` property that is itself
  an `object` with a `code` matching `ECONNREFUSED`/`ENOTFOUND`/`ETIMEDOUT`/`EAI_AGAIN` (the 4 REAL,
  live-verified Node `fetch()` transport-failure codes per REQ-DX-011 item 6), append that code; (b)
  ELSE IF `src/shell/http.ts`'s EXISTING, ALREADY-CORRECT `isTimeoutError(err)` returns `true` (REUSED
  verbatim, not reimplemented — it already handles the `"AbortError"`/`"TimeoutError"` name split this
  spec's earlier draft got wrong), append a timeout marker; (c) ELSE append nothing (today's exact
  `extractErrorMessage()` behavior, byte-for-byte unchanged for every non-network-failure case — this
  REQ is STRICTLY ADDITIVE to `extractErrorMessage()`'s existing output, per REQ-DX-NG-003's spirit:
  the ONLY case whose text changes is the network-failure case, which today produces zero specific
  information (`"fetch failed"` with the cause silently dropped) — there is no existing MEANINGFUL
  text being altered, only new information being added to a previously-uninformative case).
- REQ-DX-016 (closes spec-review it-1 SPEC-DX-3 — priority-order and export-source correctness):
  `src/core/errors.ts`'s `formatError()` currently computes `isPaymentError`/`isModelUnavailable`/
  `isServerError` as THREE INLINE, non-exported `const`s, checked in the REAL priority order
  `isModelUnavailable` → `isServerError` → `isPaymentError` (its actual `if`/`else if`/`else if` chain
  — verified directly in the source; NOTE this is the OPPOSITE order from how REQ-DX-011 originally
  listed `insufficient_funds` before `upstream_error`, a discrepancy this REQ corrects). THE SYSTEM
  SHALL extract these three checks into ONE NEW exported, order-preserving function (e.g.
  `classifyKnownError(message: string): "model_unavailable" | "server_error" | "payment_error" | null`,
  checked in EXACTLY that order, returning the FIRST match) that BOTH `formatError()` AND the new
  `code`-classification function (REQ-DX-014) call — so the human `message` text (still produced by
  `formatError()`) and the machine `code` field can NEVER disagree about which branch fired, because
  they are now driven by the SAME single check, not two independently-maintained copies. **Explicit
  non-conflation warning**: this NEW function is DISTINCT from the EXISTING exported
  `isPaymentRejectionError(message)` (`src/core/errors.ts:31-34`) — that function uses a narrower,
  DIFFERENT pattern set (`"insufficient"`/`"balance"`/`"rejected"` — no `402` check, no bare
  `"payment"` check) and is ALREADY used for an UNRELATED purpose (`src/commands/video.ts:77`'s
  decision of whether to attach an `altModels` hint to the human error text) — Phase 2 SHALL NOT
  substitute one for the other; the new `code` classifier requires the FULL `isPaymentError` logic
  from `formatError()` (the one with the `402` check and the `!hasStatus("500")` guard), not
  `isPaymentRejectionError`'s narrower subset.

## 3. Balance-unavailable reason (P1-5)

- REQ-DX-020: `wallet --action status --json`'s `base`/`solana` sub-objects (the EXISTING
  `{ address, balance }` shape, `src/commands/wallet.ts` lines ~155-163) SHALL gain a NEW field
  `balanceUnavailableReason`, present ONLY when that chain's `balance` is `null` — NEVER present
  alongside a real numeric balance, INCLUDING a genuine `0` (a true zero balance is NOT `null` and is
  ALREADY distinguishable from "balance unknown" in today's code; this REQ does not change that — it
  only explains WHY a `null` happened, when it happens).
- REQ-DX-021: FOR `base`, WHEN `balance` is `null`, `balanceUnavailableReason` SHALL be the literal
  string `"all_rpcs_failed"` — the ONE currently-distinguishable cause of a Base `null`, per
  `src/shell/wallet.ts`'s `getBaseUsdcBalance()` (lines 234-255): it tries all 3 configured
  `BASE_RPC_URLS` in sequence (`mainnet.base.org`, `base.llamarpc.com`, `1rpc.io/base`) and returns
  `null` ONLY when EVERY one either throws (network/timeout, caught by the bare `catch { continue; }`)
  OR returns a response `parseBaseUsdcCallResult()` cannot parse — there is NO other code path to a
  Base `null` today, so `"all_rpcs_failed"` is the ONE correct, fully-grounded value (this feature does
  NOT invent additional granularity, e.g. distinguishing which specific RPC failed or transport-vs
  -malformed-response, since today's code does not preserve that distinction per-attempt — adding such
  granularity would require NEW instrumentation beyond this feature's scope, not just a label).
- REQ-DX-022: FOR `solana`, WHEN `balance` is `null`, `balanceUnavailableReason` SHALL be the literal
  string `"solana_client_error"` — the ONE currently-distinguishable cause of a Solana `null`, per
  `getSolanaUsdcBalance()` (lines 223-227): it returns `null` ONLY when the Solana client's
  `getBalance()` call throws (caught by a bare `catch { return null; }`) — again, the codebase does not
  currently preserve finer detail than "it threw," so this is the one grounded value.
- REQ-DX-023: `wallet --action chain --json`'s TOP-LEVEL `activeBalance` field (a SEPARATE null
  -capable balance field used by the `chain` action, `src/commands/wallet.ts` line ~110-112 — distinct
  from `status`'s per-chain `base`/`solana` sub-objects) SHALL gain the SAME
  `balanceUnavailableReason` convention when `activeBalance` is `null`, using the SAME
  `"all_rpcs_failed"`/`"solana_client_error"` values keyed to whichever chain is active — for
  consistency across EVERY place this CLI can report a null balance, not just `status`.
- REQ-DX-024: The NON-`--json` human `wallet --action status` output, which today prints literally
  `"unavailable"` for a null balance (`src/commands/wallet.ts`'s template string, `` `$${baseBal !==
  null ? ... : "unavailable"}` ``), SHALL be updated to append the reason in parentheses, e.g.
  `"unavailable (all_rpcs_failed)"` — so a human running the CLI directly (not just `--json` callers)
  also sees WHY, not just THAT, the balance could not be read.
